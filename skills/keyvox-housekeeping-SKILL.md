## このスキルの中核ロジック

**「予約があった = 清掃必要」ではない**。ノーショウ（予約はあったが解錠ゼロ＝ゲスト来てない）の場合、清掃不要。判定:

```
Step 1. listReservations で本日範囲の予約取得
Step 2. 各予約について getLockHistory(該当lockId) で
        checkin..checkout 間に etype=9 (解錠)イベントがあるか確認
Step 3. 「実利用あり」のみを清掃対象とする
Step 4. 次のチェックイン時刻と現在時刻から「清掃可能時間枠」を算出
```

## シナリオ判別

| 発話パターン | シナリオ |
|---|---|
| 「今日の清掃対象」「清掃計画」 | A. 本日の清掃対象一覧 |
| 「今すぐ清掃に入れる部屋」「空いてる部屋」 | B. 今すぐ入れる部屋 |
| 「今日何件使われた？」「利用件数」 | C. 利用統計 |

## A. 本日の清掃対象一覧

### 手順
1. **placeIdを取得** (`place_list` で確認)
2. **本日チェックアウト予定の予約取得**:
   ```
   listReservations(placeId, fromDate=今日0:00 JST, toDate=今日23:59 JST,
                    page=1, count=100, total=1)
   ```
   `total=1` で キャンセル・既チェックアウト済を除外
3. **各予約のlockIdを引き当て**: `getUnits()` で unitId → lockIds 辞書化
4. **解錠履歴で実利用判定**:
   - ユニーク lockId ごとに `getLockHistory(lockId, records=50)` を呼ぶ
   - 各予約に対し `checkin ≦ etime ≦ checkout` かつ `etype="9"` のイベントが1件以上あれば「実利用あり」
5. **「実利用あり」だけ清掃対象に**:
   - ノーショウ（解錠ゼロ）は除外
6. **次のチェックインを引き当て**:
   - 同じunitIdで明日朝までの次の予約を listReservations (検索範囲拡張) で取得
   - 次の checkin 時刻と clearout 時刻の差 = 清掃可能時間枠
7. **表で報告**:
   ```
   ## 本日の清掃対象 (BCLtest)
   
   | # | 部屋 | チェックアウト | 次チェックイン | 清掃可能時間枠 | ゲスト |
   |---|------|---------------|---------------|--------------|--------|
   | 1 | 101 | 11:00 | 14:00 (今日) | 3時間 | 山田(済) |
   | 2 | 102 | 12:00 | 翌日10:00 | 22時間 | 佐藤(済) |
   | 3 | 103 | 13:00 | 15:00 (今日) | 2時間 | 鈴木(済) |
   
   合計 3件 (うちノーショウ除外 1件)
   ```

## B. 今すぐ入れる部屋

### 手順
1. **A の Step 1-5 と同じ手順で「実利用済み + チェックアウト済み」予約を取得**
2. **「今すぐ」条件でフィルタ**:
   - `現在時刻 > checkout` (チェックアウト時刻過ぎ)
   - **かつ** `次のcheckin - 現在時刻 > 清掃所要時間` (デフォルト60分、業務要件で調整)
3. **報告**:
   ```
   ## 今すぐ清掃に入れる部屋 (清掃所要時間: 60分の場合)
   
   | # | 部屋 | チェックアウト経過 | 次チェックインまで | 推奨着手期限 |
   |---|------|------------------|------------------|------------|
   | 1 | 101 | +30分 | 2h30m | あと 1h30m以内 |
   | 2 | 102 | +15分 | 21h45m | 余裕あり |
   ```
4. **件数ゼロ時**: 「現時点で清掃可能な部屋はありません」

### 清掃所要時間のデフォルト
- 部屋タイプにより異なる（business要件次第）
- 業種別目安: ホテル30〜45分 / レンタルスペース15〜30分 / 民泊60〜90分
- 業務要件として個別ヒアリングするか、固定値（例: 60分）で開始するかは要相談

---

## C. 利用統計（本日の実利用件数）

### 手順
1. **A の Step 1-4 で予約 × 解錠履歴の照合**
2. **集計**:
   - 本日の予約総数
   - 実利用件数（解錠イベントあり）
   - ノーショウ件数（解錠ゼロ）
   - チェックイン中件数（未チェックアウト）
3. **報告**:
   ```
   ## 本日(MM/DD) BCLtest 利用統計
   - 予約総数: 5件
   - 実利用: 3件 (60%)
   - ノーショウ: 1件
   - チェックイン中: 1件
   - 平均利用時間: 2h15m
   ```

## 共通: 「清掃可能時間枠」の計算

```
clean_start = checkout時刻 (またはmax(checkout, 現在時刻))
clean_end   = 次予約の checkin時刻 (なければ翌日0:00 or 翌週)
枠 = clean_end - clean_start
```

### エッジケース
- **同日にバックトゥバック予約**: チェックアウト直後にチェックイン → 枠が短い → 警告 「枠X分しかありません」
- **次予約なし**: 「翌日朝まで余裕あり」と表示
- **複数unit対応**: 各 unit について独立に計算

---

## ⚠️ 注意事項

### バーチャルロックの扱い
BCLtest のバーチャルロックは実際の解錠は発生しないため、「実利用あり/なし」判定の解錠履歴は実質的に意味を持たない。本番環境では物理ロックに解錠ログが残るため、このスキルが有効に機能する。

### タイムゾーン
- 予約の checkin/checkout, 解錠の etime はすべて UNIX秒
- 表示時のみ JST に変換（物件の `placeUtc` 参照）

### 「実利用」の定義
- デフォルト: 「予約期間中に1回以上解錠イベントがある」
- 厳密判定: 「checkin時刻から30分以内に解錠あり」(到着確認) 等の追加条件は業務要件次第

### 集計範囲
- 本日 = JST 0:00 〜 23:59 がデフォルト
- 「直近24時間」「今週」等の範囲指定はユーザー発話に応じて切り替え

## トラブルシューティング

| 症状 | 対処 |
|---|---|
| listReservations が0件 | 日付範囲のtimezone確認、total フィルタ調整 (`total=0` で全件) |
| getLockHistory が0件 | バーチャル機 or そのlockに通信ログが無い |
| 次チェックインが見つからない | 翌日以降まで検索範囲を広げる |
| 利用件数の数字が合わない | キャンセル予約の orderStateCode=`C` がフィルタから漏れていないか |
