# keyvox-onsite-support

ゲスト滞在中のトラブル対応スキル。**実物理デバイスに影響する操作（リモート解錠・PIN発行）を含むため、安全フローを最優先で設計**。

## ⚠️ 安全原則

このスキルが扱う操作は **直接物理ドアに影響する**:
- `unlock` → 実際にロックが開く
- `createLockPin` → 新しい鍵が物理ロックに配信される
- `disableLockPin` → 配信済みPINが即座に無効になる

**Claude は絶対に独断で実行しない**。すべての操作について以下のゲートを通る:

```
[1] 状況確認の質問
    ↓
[2] 対象ユニット/予約/ゲストの特定
    ↓
[3] 現在のロック状態を確認 (getLockStatus)
    ↓
[4] 何をするかオペレーターに最終確認 ★人間判断ゲート
    ↓
[5] 操作実行
    ↓
[6] 結果報告 (成功/失敗、次のアクション提案)
```

**ゲート [4] の確認文には以下を必ず含める**:
- 対象ロック・ユニット・ゲスト名
- 実行するアクション（解錠 / PIN発行 / 等）
- 物理的影響の明示（「実際にロックが開きます」等）
- 元に戻す手段の有無

## ⚠️ 環境前提・再認証・典型エラー

このスキルは **Claude.ai のカスタムコネクタ** 経由でのみ動作します。
401 / E2003 / DCR エラー時の対処、再認証の定型応答、Claude 向けガイドラインは以下に集約しています:

| 発話パターン | シナリオ |

## A. 鍵忘れ対応

**典型ケース**: ゲストが外出後にQR/PINを紛失。再送 or 一時PIN発行で対応。

### 手順
1. **状況聞き取り**:
   - 対象ゲスト（または部屋）
   - 現在の鍵手段（QR/PIN/カード）
   - 紛失か、QRが開けないかの区別
2. **対象予約特定**:
   - `listReservations` + searchWord で予約絞り込み
   - `getReservation(placeId, orderId)` で `unitPinList.qrUrl` / `pinCode` を取得
3. **最初に試す: 既存PINの再送**:
   - 既存の `unitPinList[0].pinCode` をオペレーターに提示
   - 「これをゲストに再送してください」と案内
   - 既存PINが有効なら新規発行不要
4. **既存PINが使えない/期限切れの場合**: 新規一時PIN発行
   - **ロック状態確認**: `getLockStatus(lockId)` で生存確認
   - **オペレーター最終確認**:
     ```
     以下で新規PINを発行します:
     - ゲスト: 山田太郎 (予約QWJX...)
     - 部屋: 101号室
     - 有効期間: 今すぐ〜18:00 (4時間)
     - 既存PINは併存（無効化はオプション）
     
     新規PINがロックに配信されます。実行してよいですか？
     ```
   - `createLockPin(unitId, sTime, eTime, targetName)` 実行
   - レスポンスの `pinCode`, `qrCode`, `shortQrUrl` をオペレーターに報告
5. **PIN配信ステータス確認** (オプション):
   - `getLockPinStatus(pinId)` を呼ぶ
   - `status: "20"` 配信待ち、`"3X"` (30/31/32) で配信完了
   - 配信完了までおよそ5分（物理ロック）
6. **ゲストへの送付方法を提案**:
   - SMS / LINE / メールなど。送付はオペレーターが行う（Claudeは情報提供のみ）

### 後始末
- 新規PIN発行時、既存PINが残っている → 状況に応じて旧PINの無効化提案 (`disableLockPin(pinId)`)

## B. 閉じ込め・緊急解錠

**典型ケース**: ゲストが中で開かない / 外で締め出された。緊急性が高いのでスピード優先。

### ⚠️ 緊急時こそ手順を守る
緊急時はオペレーターも焦るが、**間違った部屋を開けるリスクを避けるためゲートは省略しない**。

### 手順
1. **状況聞き取り**:
   - 対象ゲスト / 部屋
   - 「内側から開かない」のか「外側から閉じ込められた」のか
   - 安全上の緊急度（火災・体調不良等）
2. **対象ユニット特定**:
   - `getReservation` または `listReservations` で予約とunit特定
   - `getUnits` で該当 unitId の `lockIds` 取得
3. **ロック現在状態確認**:
   - `getLockStatus(lockId)` で生存確認
   - Wi-Fi接続: `wifi="1"` が必要
   - エラー時は「物理鍵で開けてください」「KEYVOXサポートに連絡」を案内
4. **オペレーター最終確認**:
   ```
   ⚠️ リモート解錠を実行します:
   - ロック: 101号室 (lockType BCL-QR1 / lockId QRZ...)
   - ゲスト: 山田太郎 (予約QWJX...)
   - 状況: 内側から開かない
   
   実行すると即座に物理ロックが解錠されます。
   実行してよいですか？
   ```
5. **`unlock(lockId, flag="1")` 実行**:
   - `flag="1"` = 解錠
   - `flag="0"` = 施錠
6. **結果報告**:
   - 成功: 「✅ 解錠しました。ゲストの安全確認をお願いします」
   - 失敗: エラーコードを解釈して代替手段提案（下記参照）

### よくあるエラーと対処

| エラー | 意味 | 対処 |
|---|---|---|
| `E1012` | ロックへの接続不可（電池切れ / Wi-Fi切断 / バーチャル機） | 物理鍵 / 別ルート / KEYVOXサポート |
| `401` | コネクタ認証期限切れ | 再認証してリトライ |

## C. 状況確認のみ (操作なし)

**典型ケース**: 解錠せず状態だけ確認したい。「101号室ちゃんと閉まってる？」「最後にいつ開いた？」

### 手順
1. **対象ユニット特定** (B同様)
2. **2情報を取得**:
   - `getLockStatus(lockId)` → 現在状態（開閉/電池/Wi-Fi）
   - `getLockHistory(lockId, records=10)` → 直近の解錠/施錠イベント
3. **報告**:
   ```
   101号室 (lockId QRZ...) の状況:
   
   ## 現在状態
   - Wi-Fi接続: ✅ 接続中
   - バッテリー: 75%
   - 開閉状態: 施錠
   - 最終報告時刻: 12:34
   
   ## 直近の解錠履歴 (5件)
   - 11:23 by 山田太郎 (QR解錠)
   - 09:15 by API (QR解錠)
   - 昨日 22:30 by Ken Okamoto (アプリ解錠)
   - ...
   ```

### 操作はしない
このシナリオでは `unlock` / `createLockPin` 等の **書き込み系ツールは絶対呼ばない**。  
ユーザーが状況確認のつもりで話を始めても、途中で「じゃあ開けて」と切り替わったら A/B シナリオへ遷移する。

## 物理的な影響操作のテスト時注意

このスキルを実機検証する場合:
- **テスト用ユニット (BCLtest 等の検証専用環境) のみで実施**
- **バーチャルロック (物理ハードが繋がっていないテスト用ロック) では `unlock` は E1012 で失敗するのが正常** — API 呼び出しの形式が通っていることだけ確認
- 本番ロックでの `unlock` テストは **絶対NG**（ゲストの実際の扉が開く）
- `createLockPin` でテストPIN発行する場合は `eTime` を直近（数分後）に設定し、テスト後は `disableLockPin` でクリーンアップ

## トラブルシューティング

| 症状 | 対処 |
|---|---|
| `E1012` ロック接続不可 | バーチャル機 or 実機オフライン。物理鍵を案内 |
| `getLockPinStatus.status == "20"` のまま進まない | 配信に時間かかる（通常5分以内）。それ以上経過なら再発行 or サポート |
| 既存PINが「無効です」とエラー | `getReservation` で `unitPinList[].pinStatus` 確認 (`1`=有効, `4`=無効) |
| 複数lockに紐づくunitで一部だけ解錠 | unit-lock多重度の問題。`getUnits` で全lockId取得して順次解錠 |
| オペレーター承認待ちでタイムアウト | 緊急時の場合「物理鍵で対応」を即時案内し、別経路で記録 |
